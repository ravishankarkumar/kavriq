An **agent** is an autonomous system that uses an LLM as a decision-making and reasoning engine to pursue a goal through an iterative loop of observing, planning, acting, and adjusting based on results.

Unlike a standard chatbot that simply responds once, an agent:

1. **reasons** about a higher-level objective,
2. **breaks it down** into smaller steps,
3. **acts** using external tools or system capabilities,
4. **observes** the outcome and changes course when needed.

> **A useful mental model:**  
> **Agent = LLM + Planning + Tool Use + State**

Agentic AI is not just another AI buzzword. It marks a shift from systems that merely respond to instructions toward systems that can pursue goals, make decisions, and use tools in a loop. If traditional software is a machine that follows a fixed script, an agent is closer to a digital teammate: it observes, reasons, acts, and adjusts based on what happens next.

In this article, we will build a complete understanding of what an agent actually is. We will start with the intuitive definition, look at how agents evolved in AI, compare them with traditional AI systems and deterministic pipelines, examine the core agent loop, look at real-world examples, and then end with an important engineering question: **when should you not build an agent at all?**

---

## Key Terms

Before we go further, let’s define a few terms that will appear throughout this guide.

### Tools
A **tool** is an external capability that an AI system can use to interact with the world outside the prompt. Instead of only generating text, it can call tools to search the web, perform calculations, query databases, execute code, or access files. **Tools are a central idea in agentic AI, and we will explore them in detail later in this series.**

### Deterministic Pipeline
A **deterministic pipeline** is a fixed sequence of steps defined in advance by the developer. The system follows the same execution path each time, even if one of the steps uses an LLM.

### Agent Loop
An **agent loop** is an iterative cycle in which the system observes the current state, decides what to do next, takes an action, and then evaluates the result before continuing or stopping.

## The Evolution of Agents in AI

The idea of an agent is not new. It has been part of AI for decades. In the broadest sense, an agent is any system that perceives its environment and takes actions within it. Early AI research often imagined intelligent entities that could sense the world, reason about it, and act purposefully.

For a long time, however, most real-world AI systems were much narrower than that vision. They could perform useful tasks, but only inside tightly constrained settings.

That progression happened in stages:

- **Rule-based systems** followed explicit logic written by programmers. They worked well in structured environments, but broke down when the world became messy or ambiguous.
- **Machine learning systems** learned patterns from data and became much better at prediction tasks such as classification, ranking, and detection. But most of them still performed a single bounded function.
- **Generative AI systems** added a new kind of flexibility. Large language models could generate text, code, and other content from natural-language prompts, making software feel more interactive and capable.
- **Agentic AI systems** extend this further. They do not just generate responses; they can pursue goals over multiple steps, decide what to do next, use tools, inspect intermediate results, and continue until a task is complete.

This shift is what makes modern agents so important. Earlier AI systems were often powerful but narrow. Modern LLMs changed that by providing a general-purpose reasoning and coordination engine that can operate across many tasks in natural language.

That does **not** mean every LLM application is an agent. Many LLM systems are still simple prompt-response interfaces or fixed pipelines. What changed is that LLMs made it practical to build systems that can choose actions at runtime, work through multi-step problems, and coordinate tools in a flexible way.

In other words, agents did not suddenly appear out of nowhere in the LLM era. The concept existed long before. What changed is that large language models made the old idea far more practical, accessible, and powerful.

## What Actually Makes a System an Agent?

Now let’s get precise.

A system qualifies as an agent when it does all of the following:

1. **Perceives state** from its environment  
   This could be a user message, a database record, a file, a tool result, or an API response.

2. **Decides autonomously** what to do next  
   The next step is not hard-coded in advance for every case. The system chooses based on the current situation.

3. **Acts** on the environment  
   It may call a search tool, query a database, run code, send a message, or produce a final answer.

4. **Loops** until a goal is reached or a stopping condition is hit  
   Instead of one input and one output, it may go through multiple cycles of thinking and acting.

The key word here is **autonomously**.

A lot of software uses LLMs. Not all of it is agentic. If the execution graph is fixed by the developer, then it is still a pipeline, even if one step in that pipeline uses an LLM.

<Aside type="tip" title="One-line definition">
A pipeline runs steps you predefined. An agent decides, at runtime, which step should happen next.
</Aside>

## Agentic AI vs Traditional AI

Traditional AI is usually task-specific. It performs one bounded function such as classification, prediction, ranking, or generation. In most cases, the structure of the task is decided in advance.

Agentic AI is different because it is **goal-oriented** rather than merely **task-oriented**. Instead of only answering a single prompt, it can decide how to achieve a broader objective.

Here is the difference in a compact form:

| Aspect | Traditional AI | Agentic AI |
|---|---|---|
| **Focus** | Task-specific automation | Goal-oriented autonomy |
| **Decision-making** | Fixed rules or single-pass prediction | Dynamic reasoning with intermediate decisions |
| **Human involvement** | High oversight | Lower oversight, more independence |
| **Adaptability** | Limited to predefined flow or training scope | Can adapt based on feedback and tool results |
| **Examples** | Spam filter, image classifier, prompt-response chatbot | Research assistant, coding agent, multi-step workflow agent |

A spam filter is not an agent. A classifier is not an agent. A summarizer is not an agent. Even a chatbot is usually not an agent unless it is deciding what tools to use, when to use them, and whether to continue iterating before replying.

## Deterministic Pipelines vs Autonomous Agent Loops

This is the sharpest technical distinction, and it matters a lot in practice.

Suppose the task is:

> Given a user’s research question, search the web, extract useful information, and produce a summary.

### Approach A — Deterministic Pipeline

```text
[User Query] → [Search Tool] → [Extract Top URLs] → [Scrape Pages] → [LLM Summarize] → [Output]
````

This is **not** an agent. It is a pipeline.

Why? Because the full execution graph is already determined by you, the developer. The system does not decide whether to search, how many times to search, whether the results are enough, or whether it should try a different approach. It just follows a fixed recipe.

### Approach B — Autonomous Agent Loop

```text
[User Query] → [LLM decides next step]
   • Needs more information? → Call search tool
   • Search results weak? → Refine search and try again
   • Enough evidence gathered? → Write summary
   • Goal complete? → Stop
```

This **is** an agent.

The difference is not that it uses an LLM. Both systems might use one. The difference is that the LLM is now functioning as a **decision engine** inside a loop.

## The Core Components of an Agent

A useful way to understand agents is through the basic cycle they follow. Different frameworks name the stages differently, but the underlying pattern is consistent.

## Perception

Perception is how the agent gathers information from the world.

This may include:

* user input
* system state
* files and documents
* database results
* API responses
* browser or search outputs
* multimodal inputs such as text, images, or audio

Perception gives the agent the current state it must reason about.

## Reasoning

Reasoning is where the agent interprets the state and decides what matters.

This may involve:

* identifying the real goal
* decomposing a task into subproblems
* comparing options
* deciding which tool to call
* deciding whether more information is needed
* choosing whether to stop or continue

In modern agentic systems, the reasoning engine is often an LLM, sometimes combined with memory, planning logic, or structured control policies.

## Action

Action is the part where the agent does something.

Examples include:

* calling a web search API
* running a calculator
* executing code
* querying a database
* drafting an email
* modifying a file
* sending output to another system

This is what makes an agent more than a language model. It is not just generating text; it is interacting with tools and changing state.

## Reflection

Reflection is the feedback stage.

After taking an action, the agent evaluates what happened:

* Did the action succeed?
* Did the result help achieve the goal?
* Was the retrieved information relevant?
* Did the previous plan fail?
* Should the agent retry, revise, or stop?

Reflection is one of the most important ingredients in practical agents because real environments are noisy and imperfect. Tool calls fail. Search results are weak. Code has bugs. Reflection lets the system recover rather than blindly continue.

---

## The Full Agent Cycle: Observe → Reason → Plan → Act → Reflect

The components above describe **what an agent is made of**. Now let’s look at **how those components operate over time**.

You can think of a production-grade agent as running this loop:

1. **Observe** the current state
2. **Reason** about the goal and the available evidence
3. **Plan** the next step or sequence of steps
4. **Act** using tools or outputs
5. **Reflect** on the result
6. Repeat until done

This cycle is what makes agentic systems adaptive rather than purely reactive.

A very small agent may compress some of these steps together. A more advanced one may separate them into dedicated components such as memory, planner, executor, and evaluator. But the pattern is the same.

>Problem statement: Count the number of files in the current directory, and tell an interesting fact about that number.

```python
import os
from pathlib import Path
from dotenv import load_dotenv
from google import genai
from google.genai import types

# -----------------------------
# Load environment variables
# -----------------------------
env_path = Path(__file__).parent.parent.parent / ".env"
load_dotenv(dotenv_path=env_path)

api_key = os.getenv("GEMINI_API_KEY")
if not api_key:
    raise ValueError("GEMINI_API_KEY not found in environment variables.")

client = genai.Client(api_key=api_key)

# -----------------------------
# Tool: Count regular files
# -----------------------------
def count_files(directory: str = ".") -> str:
    try:
        path = Path(directory)
        if not path.exists():
            return f"Error: directory '{directory}' does not exist."
        if not path.is_dir():
            return f"Error: '{directory}' is not a directory."
        count = sum(1 for item in path.iterdir() if item.is_file() and not item.name.startswith("."))
        return str(count)
    except Exception as e:
        return f"Error counting files: {e}"


def run_agent(user_query: str):
    chat = client.chats.create(
        model="gemini-2.5-flash",
        config=types.GenerateContentConfig(
            tools=[count_files], # Just pass the function itself!
            temperature=0.1,
            system_instruction=(
              "You are a helpful assistant. "
              "When the user asks about the number of files in a folder, "
              "use the count_files tool instead of guessing."
            ),
        )
    )
    response = chat.send_message(user_query)
    return response.text


if __name__ == "__main__":
    query = "Count the number of files in the current folder, and tell me one interesting mathematical fact about that number."
    answer = run_agent(query)

    print("\nFinal answer:\n")
    print(answer)
```

```rust
use dotenvy::from_path;
use gemini_client_api::gemini::types::request::Tool;
use std::env;
use std::fs;
use std::path::Path;

use gemini_client_api::gemini::utils::execute_function_calls;
use gemini_client_api::gemini::ask::Gemini;
use gemini_client_api::gemini::types::sessions::Session;
use gemini_client_api::serde_json::{json, Value};

#[allow(non_camel_case_types)]
pub struct count_files;

impl count_files {
    // Change the return type to Result<Value, String>
    pub async fn execute(directory: Value) -> Result<Value, String> {
        // Extract the string, default to current directory if not provided
        let directory_str = directory.as_str().unwrap_or(".");

        let path = Path::new(directory_str);

        if !path.exists() {
            // We return Ok here so the AI receives the message about the missing folder
            return Ok(json!({
                "error": format!("directory '{}' does not exist", directory_str)
            }));
        }

        let count = match fs::read_dir(path) {
            Ok(entries) => entries
                .filter_map(|entry| entry.ok())
                .filter(|entry| {
                    let name = entry.file_name();
                    let name = name.to_string_lossy();
                    entry.path().is_file() && !name.starts_with('.')
                })
                .count(),
            Err(e) => {
                return Ok(json!({
                    "error": format!("failed to read directory: {}", e)
                }));
            }
        };

        // Wrap the final JSON in Ok()
        Ok(json!({
            "directory": directory_str,
            "count": count
        }))
    }
}

fn get_count_files_schema() -> Value {
    json!({
        "name": "count_files",
        "description": "Count the number of regular non-hidden files in a directory.",
        "parameters": {
            "type": "OBJECT",
            "properties": {
                "directory": {
                    "type": "STRING",
                    "description": "Directory path. Use '.' for the current folder."
                }
            },
            "required": ["directory"]
        }
    })
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    from_path("../.env").ok();

    let api_key = env::var("GEMINI_API_KEY")
        .expect("GEMINI_API_KEY must be set in the workspace root .env");

    let mut session = Session::new(10);

    let ai = Gemini::new(
        api_key,
        "gemini-2.5-flash", 
        Some("You are a helpful assistant. Use tools when asked about files.".into()),
    )
    .set_tools(vec![Tool::FunctionDeclarations(vec![get_count_files_schema()])]);

    let prompt = "Count the number of files in the current folder, and tell me one interesting mathematical fact about that number.";
    println!("User: {}\n", prompt);

    let mut response = ai.ask(session.ask(prompt)).await?;

    loop {
        if response.get_chat().has_function_call() {
            println!("Gemini requested a function call...");

            // The macro now finds count_files::execute()
            let results = execute_function_calls!(session, count_files);

            for (idx, res) in results.iter().enumerate() {
                if let Some(r) = res {
                    println!("  Call #{} result: {:?}", idx + 1, r);
                }
            }

            response = ai.ask(&mut session).await?;
        } else {
            println!("\nGemini: {}", response.get_chat().get_text_no_think("\n"));
            break;
        }
    }

    Ok(())
}
```

## Explanation of the Code Example

This example shows the **smallest useful form of an agent**.

The model is given a tool called `count_files`, and the user asks a question that requires real information from the environment. Instead of guessing, the model decides to call the tool, reads the result, and then produces the final answer.

So the flow is:

```text
User request → model decides to use tool → tool returns result → model answers
```

The important idea is that the model is not doing the file counting itself. The **tool** does the real-world operation, while the **model** decides **when** to use it and **how** to respond.

The Python version hides much of the internal complexity, so it feels simpler. The Rust version makes the loop more explicit by showing the function schema, tool execution, and repeated checking for tool calls. Even though the syntax differs, both examples demonstrate the same agent pattern.

---

## Concepts Covered

This example introduces a few core ideas:

* **Tool use** — the model can call external functions instead of relying only on text generation.
* **Grounding** — the answer is based on real data from the environment, not a guess.
* **Function calling** — the model uses a structured interface to request tool execution.
* **Agent loop** — the system observes the task, takes an action, gets a result, and then responds.
* **Separation of roles** — the tool performs the operation, while the model handles reasoning and communication.

Even though this is a very small example, it already captures the basic structure behind many real agent systems.

## Real-World Examples of Agents

The easiest way to make this concrete is to look at real systems.

## AutoGPT

AutoGPT became one of the earliest widely discussed examples of an autonomous LLM-based agent. Its central idea was simple but powerful: give the model a high-level goal, let it break that goal into substeps, call tools, and keep iterating.

It was important historically because it showed people what agent loops looked like in practice. It also exposed the limitations quickly: drift, wasted steps, fragile planning, and the need for human checkpoints.

## Devin

Devin is a coding-focused agent designed to operate more like an autonomous software engineer. It can inspect codebases, write code, run tests, debug issues, and propose changes over multiple steps.

This is a genuinely agentic setting because the exact path cannot be known in advance. Debugging is inherently iterative. The system must inspect failures, revise its plan, try again, and stop only when the task is complete or blocked.

---

## Multi-Agent Systems

A multi-agent system uses multiple agents with different roles that coordinate with each other.

Examples include:

* a researcher agent that gathers information
* a planner agent that breaks work into subtasks
* a coder agent that implements solutions
* a reviewer agent that checks quality
* a coordinator agent that decides what happens next

These architectures are useful when a task is too complex for a single monolithic loop, or when role separation improves reliability.

## When NOT to Use an Agent

This is one of the most important engineering lessons in agentic AI.

Agents are powerful, but they are not free. They introduce more latency, more cost, more unpredictability, and more failure modes. A lot of teams build agents where a simple pipeline would have been faster, cheaper, and more reliable.

Here is a practical comparison:

| Use Case                                                  | Pipeline or Agent? | Reason                                      |
| --------------------------------------------------------- | ------------------ | ------------------------------------------- |
| PDF → extract text → summarize                            | ✅ Pipeline         | Fixed sequence of steps                     |
| Classify a support ticket                                 | ✅ Pipeline         | Single-pass decision                        |
| Generate a weekly report from known tables                | ✅ Pipeline         | Fully predictable flow                      |
| Research assistant that refines searches                  | 🤖 Agent           | Number and type of searches unknown upfront |
| Debug code with repeated test runs                        | 🤖 Agent           | Requires iteration and feedback             |
| Customer service flow with booking, refunding, escalation | 🤖 Agent           | Next step depends on context and outcomes   |

The rule of thumb is simple:

This single distinction will save you a lot of engineering pain.

## The Cost of Autonomy

Autonomy sounds exciting, but it comes with real trade-offs.

Each turn of the agent loop may require another model call. That means:

* **Higher latency**
  A pipeline may require one call. An agent may require five.

* **Higher cost**
  More steps usually mean more tokens and more tool usage.

* **Lower predictability**
  The agent may choose a path you did not anticipate.

* **More failure modes**
  It may call the wrong tool, misread a tool result, repeat unnecessary steps, or stop too early.

That is why production agent design is not just about “making it autonomous.” It is about carefully deciding where autonomy is actually worth the overhead.

In many systems, the best architecture is hybrid:

* use deterministic pipelines where the flow is obvious
* use agents only where runtime decisions are genuinely necessary

That usually gives you the best trade-off between power and reliability.

---

## Why Agents Matter

Despite the trade-offs, agents represent an important shift in AI system design.

They allow us to move from:

* **single-shot responses** to **multi-step task completion**
* **static prompt-response behavior** to **dynamic tool-using workflows**
* **reactive assistants** to **goal-directed systems**

This is what makes agentic AI so relevant for real applications such as:

* software engineering assistants
* research systems
* workflow automation
* business operations
* customer support orchestration
* scientific and data-analysis workflows

Agents matter because the real world is rarely a one-step problem.

---

## Wrapping Up

An agent is not just an LLM with a fancy prompt. It is a system that perceives state, decides what to do next, acts using tools or outputs, and loops until it reaches a goal or hits a stopping condition.

That means:

* not every LLM app is an agent
* pipelines and agents are fundamentally different execution models
* the agent loop is the real core of agentic systems
* reflection and iteration are what make agents adaptive
* sometimes the smartest choice is *not* to build an agent at all

If you understand that distinction clearly, you already have a much stronger foundation than most beginner discussions of agentic AI provide.
