# The Planner / Reasoner

When a user gives an agent a task, the instruction is usually **high level**.

Examples:

* “Summarize the latest research on fusion energy.”
* “Find the cheapest flights to Tokyo next month.”
* “Compare the performance of RTX 4090 and H100 for machine learning.”

These instructions do **not specify how the task should be performed**.

The agent must determine the steps required to complete the task.

This is the responsibility of the **planner / reasoner**.

---

# What the Planner Does

The planner converts a **goal** into a **sequence of actions**.

Conceptually:

```
Goal
 ↓
Reasoning
 ↓
Plan
 ↓
Actions
```

For example:

```
Goal: Compare GPUs for machine learning

Plan:
1. Search for benchmark data
2. Extract key metrics
3. Compare training performance
4. Summarize results
```

The planner therefore acts like the **decision engine** of the agent.

---

# The Difference Between Reasoning and Planning

The terms **reasoning** and **planning** are often used interchangeably, but they refer to different concepts.

| Concept   | Description                        |
| --------- | ---------------------------------- |
| Reasoning | Thinking about the problem         |
| Planning  | Constructing a sequence of actions |

Example:

```
Reasoning:
"I need benchmark data to compare GPUs."

Planning:
Step 1 → Search benchmark results  
Step 2 → Extract metrics  
Step 3 → Compare GPUs
```

Reasoning generates **insight**.

Planning generates **structure**.

Both are necessary for intelligent agents.

---

# Where Planning Happens in the Agent Loop

Recall the full agent loop:

```
observe → reason → plan → act → reflect
```

The planner sits between **reasoning and execution**.

```
Observation
   ↓
Reasoning (LLM)
   ↓
Planning
   ↓
Tool Execution
```

Without planning, an agent may jump directly from reasoning to action.

This often leads to **inefficient or chaotic behavior**.

---

# Turning Goals Into Plans

Let us examine a simple planning example.

User request:

> “Research the economic impact of electric vehicles.”

A reasonable plan might look like this:

```
1. Search for EV market statistics
2. Identify economic indicators
3. Analyze industry trends
4. Summarize findings
```

This decomposition allows the agent to solve the problem **step by step**.

---

# Planning Depth

Planning strategies vary in complexity.

Agents may use:

| Planning Type         | Description                      |
| --------------------- | -------------------------------- |
| Single-step planning  | Decide next action only          |
| Linear planning       | Create sequential plan           |
| Hierarchical planning | Break tasks into sub-tasks       |
| Tree search planning  | Explore multiple reasoning paths |

Most modern agents combine **linear planning with iterative reasoning**.

---

# Task Decomposition

Task decomposition is the process of splitting a complex goal into smaller tasks.

```
Goal
 ↓
Subtasks
 ↓
Actions
```

Example:

```
Goal: Write report on AI chip market

Subtasks:
• Identify major companies
• Gather market size data
• Compare architectures
• Summarize trends
```

This approach improves:

* reasoning accuracy
* tool usage
* reliability of the agent

---

# Example Decomposition

Below is a possible decomposition produced by an LLM.

```
Goal: Compare Python and Rust for AI systems

Plan:

1. Identify performance benchmarks
2. Compare ecosystem maturity
3. Analyze developer productivity
4. Summarize tradeoffs
```

Each step becomes an **actionable unit** for the runtime.

# Implementing a Simple Planner

Let us implement a minimal planner that generates a plan using an LLM.

<Tabs>

<TabItem label="Python" icon="seti:python">

```python
def generate_plan(goal, llm):

    prompt = f"""
    Break the following goal into a sequence of steps.

    Goal: {goal}

    Return a numbered plan.
    """

    response = llm.generate(prompt)

    return response
```

</TabItem>

<TabItem label="Rust" icon="seti:rust">

```rust
fn generate_plan(goal: &str, llm: &LLMClient) -> String {

    let prompt = format!(
        "Break the goal into steps:\nGoal: {}",
        goal
    );

    llm.generate(&prompt)
}
```

</TabItem>

</Tabs>

The plan then becomes part of the **agent's working memory**.

# Reactive Agents vs Planning Agents

Not all agents explicitly generate plans.

There are two common approaches.

---

# Reactive Agents

Reactive agents operate using **local decision making**.

```
Observation → Action
```

They decide only the **next step**, not the entire plan.

Example loop:

```
observe
reason
act
observe
reason
act
```

Advantages:

* simple architecture
* low latency
* flexible behavior

Disadvantages:

* inefficient for complex tasks
* may repeat steps
* harder to control

---

# Planning Agents

Planning agents generate a **global plan** before execution.

```
Goal
 ↓
Plan
 ↓
Execute Steps
```

Advantages:

* structured problem solving
* more predictable behavior
* easier debugging

Disadvantages:

* planning may be expensive
* plans may become outdated

---

# Hybrid Planning

Most modern agents combine both approaches.

```
High-level plan
     ↓
Reactive execution
     ↓
Re-plan if necessary
```

Example:

```
Plan:
1. Search papers
2. Read top 3 papers
3. Summarize findings
```

If the search fails, the agent may **re-plan** dynamically.

---

# Planning in Modern Agent Frameworks

Many agent frameworks include built-in planning capabilities.

Examples:

| Framework | Planning Strategy       |
| --------- | ----------------------- |
| LangChain | tool-based reasoning    |
| LangGraph | state graph planning    |
| AutoGen   | conversational planning |
| CrewAI    | role-based planning     |

Despite their differences, they all implement the same core idea:

> Convert goals into **structured sequences of actions**.

---

# Why Planning Matters

Planning dramatically improves agent performance.

Benefits include:

### Better Tool Usage

The agent knows **which tools to use and when**.

---

### Reduced Hallucination

Planning forces the model to **think before acting**.

---

### More Reliable Execution

Complex tasks become manageable through decomposition.

# A Planning Example in Action

Below is what a reasoning sequence might look like.

```
Goal: Find best ML GPU

Thought: Need benchmark data
Action: web_search("ML GPU benchmark")

Observation: RTX 4090 and H100 results

Thought: Compare training speed
Action: analyze_benchmarks()

Observation: H100 significantly faster

Thought: Summarize results
```

Here the planner guides the **sequence of steps**.

---

# The Planner as the Brain of the Agent

If the **LLM is the CPU** and the **runtime is the operating system**, then the planner is the **executive function**.

It determines:

* what to do next
* which tools to use
* how to break down problems
* when to stop

Without planning, agents become **directionless**.
