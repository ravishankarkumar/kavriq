# Trajectory Evaluation

An **agent trajectory** represents the sequence of steps taken by an agent while solving a task.

Example trajectory:

```text
User Question
   ↓
Thought: search for GPU benchmarks
   ↓
Action: web_search("H100 vs RTX 4090 benchmarks")
   ↓
Observation: benchmark results
   ↓
Thought: compare training throughput
   ↓
Final Answer
```

Each step contains valuable information about **how the agent reasoned**.

Trajectory evaluation measures the **quality and efficiency of these steps**.

---

# Why Trajectories Matter

Two agents may produce the same final answer but follow very different reasoning paths.

Example:

Agent A:

```text
Search → retrieve relevant source → answer
```

Agent B:

```text
Search → search again → random guess → answer
```

Both agents produce the correct answer.

However, Agent A demonstrates **strong reasoning**, while Agent B appears unreliable.

Trajectory evaluation helps identify these differences.

---

# Dimensions of Trajectory Evaluation

Several metrics can be used to evaluate trajectories.

Common dimensions include:

| Metric Category   | Purpose                    |
| ----------------- | -------------------------- |
| efficiency        | measure number of steps    |
| tool selection    | evaluate tool usage        |
| reasoning quality | assess logical consistency |

These metrics provide insight into **how the agent solves problems**, not just whether it succeeds.

---

# Efficiency Metrics

Efficiency metrics measure how effectively an agent solves tasks.

Example questions:

* How many steps were required?
* How many tool calls were made?
* How long did the task take?

Example trajectories:

Efficient agent:

```text
Search → analyze → answer
```

Inefficient agent:

```text
Search → search again → search again → analyze → answer
```

Even if both succeed, the second agent wastes resources.

---

# Measuring Step Count

A simple efficiency metric is **trajectory length**.

Example:

```text
Trajectory length = number of reasoning + action steps
```

Example comparison:

| Agent   | Steps |
| ------- | ----- |
| Agent A | 3     |
| Agent B | 7     |

Shorter trajectories often indicate **more efficient reasoning**.

However, extremely short trajectories may indicate **guessing**, so efficiency must be interpreted carefully.

---

# Tool Selection Metrics

Agents often rely on tools such as:

* web search
* databases
* APIs
* code execution environments

Evaluating tool usage helps determine whether the agent used **appropriate tools**.

Example scenario:

Question:

```text
What is the market share of NVIDIA GPUs?
```

Correct tool usage:

```text
web_search("NVIDIA GPU market share")
```

Incorrect tool usage:

```text
calculator(80 + 15)
```

Tool selection metrics measure whether the agent chose **the correct tool for the task**.

---

# Tool Usage Efficiency

Agents sometimes overuse tools.

Example trajectory:

```text
Search → search again → search again → answer
```

Evaluation metric:

```text
tool_calls_per_task
```

Lower values often indicate **better planning**.

However, complex tasks may legitimately require multiple tool calls.

---

# Tool Accuracy

Another important metric is **tool accuracy**.

Example evaluation:

```text
Correct tool selected: web_search
Incorrect tool selected: database_query
```

Tool accuracy measures how often the agent chooses **the appropriate tool**.

---

# Reasoning Quality Metrics

Trajectory evaluation also measures the **quality of reasoning steps**.

Example reasoning step:

```text
Thought: GPUs accelerate training because they perform parallel matrix operations.
```

Evaluation criteria may include:

* logical consistency
* factual correctness
* clarity of reasoning

Poor reasoning example:

```text
Thought: GPUs are faster because they are expensive.
```

This reasoning is incorrect.

---

# Evaluating Reasoning with LLM Judges

Reasoning quality is often evaluated using **LLM judges**.

Example prompt:

```text
Evaluate the reasoning step:

"GPUs accelerate deep learning because they perform parallel matrix operations."

Is this reasoning correct?
```

Example output:

```text
Score: 5
Reason: Accurate explanation of GPU parallelism.
```

This approach allows reasoning quality to be evaluated automatically.

---

# Error Detection in Trajectories

Trajectory evaluation also helps identify **failure modes**.

Example issues include:

| Failure Type           | Description            |
| ---------------------- | ---------------------- |
| hallucinated reasoning | incorrect assumptions  |
| tool misuse            | calling the wrong tool |
| redundant actions      | repeated tool calls    |
| incomplete reasoning   | missing logical steps  |

Identifying these patterns helps developers improve agent design.

Conceptually:

Agent execution

      ↓
Performance report

Each trajectory is analyzed to produce evaluation scores.

# Example Implementation

<Tabs>

<TabItem label="Python" icon="seti:python">

```python
def evaluate_trajectory(trajectory):

    steps = len(trajectory)

    tool_calls = sum(1 for s in trajectory if s["type"] == "tool")

    return {
        "trajectory_length": steps,
        "tool_calls": tool_calls
    }
```

</TabItem>

<TabItem label="Rust" icon="seti:rust">

```rust
fn evaluate_trajectory(traj: &Vec<Step>) -> Metrics {

    let steps = traj.len();

    let tool_calls = traj.iter()
        .filter(|s| s.kind == StepKind::Tool)
        .count();

    Metrics { steps, tool_calls }
}
```

This provides basic efficiency metrics.

# Combining Multiple Metrics

In practice, systems combine several metrics into an overall evaluation score.

Example:

```text
Final Score =
0.4 × task success
0.3 × reasoning quality
0.2 × tool accuracy
0.1 × efficiency
```

This weighted scoring approach provides a **balanced evaluation**.

---

# Trajectory Evaluation in Production Systems

Many production AI systems track trajectory metrics.

Examples include:

| System             | Metrics Tracked          |
| ------------------ | ------------------------ |
| research agents    | search efficiency        |
| coding agents      | tool usage accuracy      |
| enterprise agents  | workflow completion rate |
| autonomous systems | decision correctness     |

These metrics help teams continuously **improve agent behavior**.

---

# Trajectory Evaluation as Observability

Trajectory analysis also functions as **observability for agents**.

Developers can inspect trajectories to understand:

* why an agent failed
* where reasoning broke down
* which tools were misused

This is similar to **debugging logs in traditional software**.
